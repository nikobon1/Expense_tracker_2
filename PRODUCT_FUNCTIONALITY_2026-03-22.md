## 1. Веб-приложение

### 1.1 Аутентификация

- Вход через Google OAuth.
- Защищенный login-flow для production.
- Усиленный dev/staging login:
  - `DEV_LOGIN_ENABLED=true` больше недостаточно само по себе.
  - Нужен непустой `DEV_LOGIN_PASSWORD`, иначе credentials-provider не включается.

### 1.2 Добавление покупок

- Загрузка фото чека через web UI.
- AI-распознавание чека.
- Сохранение чека и позиций в базу.
- Web `Quick Add` / быстрое добавление покупки без фото чека.
- Поддержка ручного ввода магазина, даты и общей суммы.

### 1.3 Редактирование чеков

- Открытие чека из dashboard.
- Редактирование:

  - позиций
  - категорий
  - цен
- Удаление чека.
- Сравнение существующего чека с новым фото.
- Сравнение чека с изображением из Telegram.
- Применение результатов повторного AI-анализа поверх существующего чека.

### 1.4 Категории

- Поддержка пользовательских категорий.
- Добавление новых категорий из интерфейса.
- Удаление пользовательских категорий.
- Нормализация сломанных / битых значений категорий в API и при сохранении.
- Расширение набора категорий, включая travel/lifestyle additions.

## 3. Telegram bot

### 3.1 Базовый flow

- Приём фото чека в Telegram.
- AI-распознавание чека.
- Сохранение результата в общую БД.
- Дедупликация Telegram updates (`telegram_processed_updates`).
- Черновики чеков и ручных покупок (`telegram_receipt_drafts`).
- Allowlist пользователей через `TELEGRAM_ALLOWED_USER_IDS`.

### 3.2 Ручное добавление покупок в Telegram

- Команда `/manual`.
- Быстрый seed-формат `/manual Store; 12.49; 14/02/26`.
- Guided manual flow:
  - amount -> store -> date -> save
- Быстрый путь `/manual 12.49`.
- Поддержка plain numeric message как суммы без префикса `Sum`.
- Draft editing commands:
  - store
  - sum
  - date
  - item
  - save
- Защита от сохранения неполного manual draft.
- `Today` корректно завершает шаг ввода даты.

### 3.3 Telegram UX

- Reply keyboard main menu.
- Fallback от inline-only меню к reply keyboard из-за нестабильного поведения inline buttons в реальном Telegram client.
- Сохранены callback handlers как fallback.
- Обработка текстовых аналогов для тех же действий.
- Команда `/start` для обновления актуальной клавиатуры.
- Русский текстовый UX без mojibake.
- Английские command aliases как fallback.

### 3.4 Улучшения парсинга в Telegram

- Гибкий парсинг сумм:
  - `12.49`
  - `12,49`
  - `1 234,56`
  - суммы с символами валют
- Гибкий парсинг дат:
  - `14/02/26`
  - `14-02-2026`
  - `2026-02-14`
  - compact digits
  - `Сегодня`
- Поддержка custom item naming в manual draft.
- Extended `/manual` seed format с optional item name.
- Многострочные команды в одном сообщении:
  - обработка line-by-line
  - одно итоговое сообщение вместо серии промежуточных ответов

## 4. База данных и backend

### 4.1 Работа с БД

- Общая база данных для web и Telegram.
- Atomic receipt writes в Postgres:
  - insert receipt + items в одной транзакции
  - update receipt + replace items в одной транзакции

### 4.2 Миграции и схема

- Удалено runtime schema DDL из API-paths.
- Добавлен явный migration script:
  - `npm run db:migrate`
- При отсутствии схемы API теперь возвращает понятный `503`, а не пытается молча создавать таблицы на лету.

### 4.3 API / backend endpoints

- API для сохранения расходов.
- API для чтения и редактирования чеков.
- API webhook для Telegram.
- Health endpoint.

### 4.4 OCR / receipt analysis

- Улучшено распознавание total:
  - `Total`
  - `TOTAL A PAGAR`
  - `Total a pagar`
- Снижен риск выбора `subtotal` вместо финальной суммы.

## 5. Качество и устойчивость продукта

- Исправления битой кодировки в web и Telegram текстах.
- Repo-level UTF-8 editor defaults, чтобы снизить риск новых encoding regressions.
- Trim Telegram bot token при загрузке из env, чтобы убрать ошибки из-за trailing whitespace.
- Более предсказуемое локальное / staging тестирование.

6. Смотреть аналитику по категориям и дням.
7. Открывать чек, редактировать его и пересравнивать с новым фото.
8. Экспортировать расходы.
9. Работать с мобильным dashboard-экраном с улучшенным UX и русской локализацией.
